Estrutura básica

Uma vez colocados os arquivos base do sistema em um diretório para iniciar o desenvolvimento, basta criar os controllers, models, views e arquivos como CSS e JavaScript, armazenando-os nas respectivas pastas do sistema para começar a criar algo incrível.

A estrutura básica não é muito grande ou complexa. Ainda assim, merece atenção para que cada componente esteja em seu devido lugar durante o desenvolvimento da aplicação.

A relação dos diretórios padrão é a seguinte, podendo diferir dos de diretórios e arquivos baixados do repositório, mas contendo o essencial e ideal para a execução do sistema:

Nomenclaturas

O padrão de nomes adotado define que todo e qualquer controller deve possuir a primeira letra de seu nome maiúscula e que deve terminar com o termo "Controller" - sem aspas e também com a letra C maiúscula. Isso se aplica tanto ao arquivo quanto ao nome da classe.

Portanto, para criar um controller que seria originalmente chamado de, por exemplo, "usuario", o nome do arquivo e da classe deve ser "UsuarioController".

Para toda e qualquer classe é recomendado definir seu nome com a primeira letra maiúscula, em sintonia com o padrão dos controllers. Tanto para o arquivo quanto para o nome da classe.

Para views, é recomendado utilizar a primeira letra maiúscula no nome do arquivo.

No banco de dados, cada tabela deve ter exatamente o mesmo nome dos models equivalentes, sem a necessidade de serem idênticos quanto às letras maiúsculas e minúsculas.

Quanto aos atributos das tabelas, o padrão adotado para a chave primária é "id".

Observação: ao realizar uma chamada aos arquivos, como, por exemplo, pelas funções load(), view() ou model(), utilize o nome dos arquivos exatamente como foram definidos (com as letras maiúsculas e minúsculas), para evitar erros ao incluí-los.


Constantes do sistema

O arquivo /app/system/SystemData.php contém as constantes que são utilizadas pelo sistema na configuração de determinados recursos.

Estas constantes devem ser alteradas de acordo com o projeto, definindo os parâmetros de configuração com o banco de dados, as configurações do sistema de e-mails, dados referentes à sessão criada pelo sistema e informações sobre o caminho da raiz do projeto.

Seguem maiores detalhes sobre as constantes do sistema que devem ser alteradas de acordo com o projeto:

Const. Base

 
define('SYSROOT', '/'); // Define a raiz da aplicação em relação à raiz do servidor

Const. Sessão

 
define('SESSION_NAME', 'Application'); // Define o nome da sessão
define('SESSION_HTTP_ONLY', 1); // Propriedade HTTP_Only dos cookies de sessão (1 = TRUE, 0 = FALSE)
define('SESSION_ACTIVITY_TIME', 5); // Tempo de inatividade até que seja realizado o logout do usuário (0 = não faz o logout automático)

Const. Banco de Dados

 
define('DB_HOST', 'localhost'); // Endereço do host do banco de dados
define('DB_USER', 'user'); // Nome de usuário no banco de dados
define('DB_PASS', '123'); // Senha do usuário no banco de dados
define('DB_NAME', 'MyDatabase'); // Nome do banco de dados a ser utilizado

Base de conexão ao banco de dados: PDO. Certifique-se de tê-lo habilitado.


Const. E-mail

 
define('MAIL_FROM_NAME', 'David'); // Atalho para o nome a ser exibido como remetente do e-mail

define('SMTP_HOST', 'smtp.email.com'); // Endereço do servidor SMTP
define('SMTP_PORT', '465'); // Porta do servidor SMTP
define('SMTP_USER', 'Username'); // Usuário no servidor SMTP
define('SMTP_PASS', 'Password'); // Senha no servidor SMTP
define('SMTP_SECURE', 'ssl'); // Tipo de segurança do servidor SMTP ('tls' ou 'ssl')
define('SMTP_AUTH', TRUE); // Define se o servidor SMTP exige autenticação

Base de envio de e-mails: PHPMailer.



Rotas

As rotas sempre encaminharão o visitante para a função "method" da classe "controller", passando os argumentos "id" e "param", quando houver, representados nos exemplos abaixo.


Rotas automáticas

As rotas são definidas automaticamente com base na URL conforme o seguinte padrão: 

http://baseurl/controller/method/id/param1/param2/...

Portanto, o primeiro parâmetro passado pela URL é referente ao controller, o segundo refere-se ao método e, a partir do terceiro parâmetro, são definidas as variáveis passadas ao sistema por meio do método GET, sendo a primeira delas referente ao ID e as demais indexadas como variáveis qualquer.


Rotas personalizadas

Podem também ser definidas rotas personalizadas ao editar o arquivo /app/system/Routes.php e adicionando a rota ao vetor $route conforme o padrão: 

$route['URL'] = 'Controller/Method';

Observe que URL deve ser definida como um caminho relativo a partir da raiz, como nos exemplos abaixo: 

$route['user/restrictedarea'] = 'ExampleController/example';
$route['/user/restrictedarea'] = 'ExampleController/example';
$route['/user/restrictedarea/'] = 'ExampleController/example';
$route['user/restrictedarea/'] = 'ExampleController/example';

Todos os exemplos representam a mesma rota.

Para direcionar uma rota ao método index de um controller, é possível definir o valor da rota como apenas o controller, sem o método. Por exemplo, ao invés de ExampleController/index, é também possível definir como ExampleController e obter o mesmo efeito.

Observação: As rotas mais abaixo no arquivo sobrescrevem as anteriores, portanto, se houver uma rota '/page/?' e, mais abaixo, uma rota '/page/newpage', a primeira será sobrescrita pela segunda e não funcionará.


Rotas com argumentos

É possível determinar quando uma rota deve receber as variáveis GET passadas como argumentos da URL adicionando um '?' (sem aspas) à rota, conforme o padrão: 

$route['posts/?'] = 'PostsController/show';

Dessa forma, todo e qualquer parte da URL vinda depois de 'posts' será traduzida para uma variável passada à função determinada na rota.

No caso de serem passados mais de um dado, o primeiro sempre será definido, na variável passada à função, com o índice 'id'. Por exemplo, a URL 'posts/12/14/feb/2017' irá resultar em um vetor semelhante a: 

$param = array(
    0 => '14',
    1 => 'feb',
    2 => '2017',
    id => '12'
);

Observe que o nome da variável é definido pelo nome do argumento recebido pelo método do controller que o receberá.

Portanto, se a função for definida como show ($get) { ... }, o nome do vetor passado será $get, e não $param.



Mensagens de erro do sistema

Por padrão, existem algumas mensagens de erro que são utilizadas nas validações realizadas pelas funções aqui descritas.

Por exemplo, quando é encontrado um erro pela função Validation::check() e são retornados, ao formulário, os erros encontrados no preenchimento dos mesmos, são essas mensagens de erro que são exibidas para o usuário.


Alterar texto das mensagens

É possível alterar o texto das mensagens, indo diretamente ao arquivo ao qual pertence a função que a utiliza.


Validação de dados

Para alterar as mensagens de validação de dados, vá até o arquivo app/system/components/Validation.php e procure pelas definições das variáveis $message.

Tais variáveis estão dispostas da seguinte maneira (o número da linha pode variar): 

 43| $message = '...'; // Caso um campo seja obrigatório e não tenha sido preenchido
 51| $message = '...'; // Caso um valor deva ser um número inteiro
 59| $message = '...'; // Caso um valor deva ser um número decimal
 67| $message = '...'; // Caso um valor deva ser composto de apenas letras e espaço
 75| $message = '...'; // Caso um valor deva ser composto de apenas letras, números e espaço
 83| $message = '...'; // Caso um valor deva ser um endereço de e-mail com sintaxe válida
 93| $message = '...'; // Caso um valor não tenha o número mínimo de caracteres, definido por $rulepart[1]
102| $message = '...'; // Caso um valor ultrapasse o número máximo de caracteres, definido por $rulepart[1]
110| $message = '...'; // Caso um campo não seja igual a um dado outro campo, definido por $rulepart[1]

Lembrando que $field, presente em todas as mensagens, representa o nome do campo do qual se trata quando a mensagem é retornada.

Por exemplo, se, ao analisar um campo nomeado username que é obrigatório e não esteja preenchido, é retornada a seguinte mensagem, sendo que $field é responsável por exibir o nome do campo: 

"username" deve ser preenchido
Autenticação de usuário

Para alterar as mensagens de validação de dados, vá até o arquivo /app/system/core/Auth.php e procure pelas definições de variáveis que contenham "Msg em seu nome.

Tais variáveis estão dispostas da seguinte maneira (o número da linha pode variar): 

 17| $noUserMsg = '...'; // Caso o usuário buscado não exista no banco de dados
 18| $wrongPassMsg = '...'; // Caso a senha digitada esteja incorreta
 81| $usedUserMsg = '...'; // Caso o usuário digitado não esteja disponível para registro


Controllers

Todos os conrollers criados devem extender o controller principal "Controller". Os controllers fazem a conexão entre os dados armazenados (obtendo-os através dos models) e sua exibição (a partir das views).

Os controllers devem ser armazenados no diretório /app/controllers e devem ter a primeira letra de seu nome maiúscula, seguida da palavra Controller. 

class SomeController extends Controller { ... }

É importante ressaltar que cada função representa o método para o qual a URL direcionará.

Assim, uma função index em um controller qualquer, será responsável por atender a URL quando um método não for definido. Por exemplo, ao entrar com a URL mysite.com/user, a função index do controller User processará a requisição (assumindo-se o uso das rotas automáticas). 

mysite.com/user // Essa URL será processada pela seguinte função:
class UserController extends Controller { 
   public function index () { ... } 
} 

mysite.com/user/register // Essa URL será processada pela seguinte função:
class UserController extends Controller { 
   public function register () { ... } 
}

Página principal

A página principal, raiz do sistema, é exibida a partir da rota padrão para o MainController, utilizando seu método index. Para alterar a página inicial, pode-se alterar a view Default ou o próprio controller, ou mesmo substituí-lo por um novo, com outro nome, e criar uma rota personalizada para a raiz do sistema, apontando para este novo controller. 

mysite.com/ // Essa URL será processada, por padrão, em:
class MainController { 
   public function index () { ... } 
}

getRequest()

Identifica o tipo de requisição realizada para a página (get, post, put, delete, etc.). 

string getRequest ( )

Usage example:
$requestType = getRequest ();

Retorna uma string, em letras minúsculas, contendo o tipo de requisição realizada.


Parâmetros GET

Considerando a passagem de parâmetros padrão das rotas com argumentos, as variáveis passadas aos métodos são processadas como trechos da URL, separadas por uma barra. Desta forma, não são utilizadas as variáveis GET com o intuito de manter a URL mais limpa.

Entretanto, para que um método do controller possa fazer uso dos valores vindos da URL, ele deve poder receber argumentos, ou seja, a função deve estar habilitada a receber parâmetros. 

public function something ( $params ) { ... }

Ao utilizar uma variável, por exemplo, $params, para receber os dados vindos da URL em um método, todos os parâmteros passados serão adicionados a ela, de forma que ela passa a ter uma estrutura semelhante à da variável $_GET. Assim, os parâmetros são alcançados utilizando $params["id"], $params["param1"], etc.

Por outro lado, os parâmetros GET ($_GET) continuam podendo ser usados, tendo à disposição apenas a função filterGet() para manuseio destes dados.

Portanto, se utilizar o parâmetro GET tradicional, a URL ficará bagunçada mas os dados serão acessados normalmente. 

URL: mysite.com/controller/method?id=X&something=Y

GET vars:
array(2) {
  ["id"]=>
    string(1) "X"
  ["something"]=>
    string(1) "Y"
}

Inclusão de componentes

load()

Carrega um arquivo, passado o caminho a partir da raiz da aplicação. Caso não consiga encontrar tal arquivo, gera um Fatal Error. 

bool load ( string $pathToFile )

Usage example:
load ('/app/lib/anything.php');

Retorna TRUE se o arquivo já tiver sido carregado ou FALSE caso contrário.


view()

Carrega uma view, dado o nome do arquivo sem a extensão. Caso não consiga encontrar tal arquivo no diretório /app/views, gera um Fatal Error.

Pode receber uma variável de dados para usar no arquivo de view, passado e acessado sempre por $data, conforme citado em passagem de dados. 

bool view ( string $viewName, [$data = NULL] )

Usage example:
view ('LoginForm');
view ('somePage', $viewData);

Retorna TRUE se o arquivo já tiver sido carregado ou FALSE caso contrário.


model()

Carrega um model, dado o nome do arquivo sem a extensão. Caso não consiga encontrar tal arquivo /app/models, gera um Fatal Error.

Por padrão todos os models são carregados automaticamente, portanto, essa função não se faz muito necessária. 

bool model ( string $viewName )

Usage example:
model ('Animal');

Retorna TRUE se o arquivo já tiver sido carregado ou FALSE caso contrário.



Views

As views são responsáveis pela exibição dos dados.

Não há um padrão definido para a forma como as views são utilizadas. Elas apenas devem ser armazenadas no diretório /app/views.


Inclusão de views

Quando for necessário usar uma view no sistema, deve-se utilizar a função view() para carregar o arquivo que contém as informações necessárias.

Também é possível passar dados para uma view, adicionando o segundo parâmetro à sua chamada. O dado passado fica acessível na view pelo nome $data. 

public function someFunction () {
...
return view('some');
return view('another', $someData);
}

Observação: utilizar o return ao carregar uma view é opcional, mas pode ser útil para garantir que não haverá código sendo executado após o carregamento da tela do usuário.


Passagem de dados

É possível passar uma variável para a view através da função view(), passando a variável desejada como segundo argumento.

Caso haja a necessidade de passar mais de uma variável para a view, é preciso colocar as múltiplas variáveis a serem passadas em um array e então passar esse array para a função view().

Note que toda view, ao receber uma variável, tem seu nome definido como $data. Assim, para acessar qualquer valor enviado para a view, deve-se usar esta variável. 

Passando um valor ou variável única:
view('someView', 'Fruits');
Na view, para se obter o valor, usa-se:
$data

Passando um array:
view('someView', [$var1, $var2, var3, ... ]);
Na view, para se obter os valores, usa-se:
$data[0] // Contém o valor de $var1
$data[1] // Contém o valor de $var2
...

Models

Todos os models criados devem extender o model principal "Model". São os models quem farão a comunicação com o banco de dados para realizar consultas e inserir, alterar ou apagar dados.

Os models devem ser armazenados no diretório /app/models. 

class SomeModel extends Model { ... }

Quando representando um objeto armazenado no banco de dados, é essencial que o model tenha exatamente o mesmo nome da tabela no banco a qual representa.

Assim, para representar uma tabela denominada, por exemplo, por animal no banco de dados, o model referente a ela deve se chamar Animal também. 

CREATE TABLE animal ( ... ); // Representa a tabela no banco de dados
class Animal extends Model { ... } // Representa o model que se comunica com a tabela

Criação de um model

Ao criar um model, é preciso adicionar a ele variáveis privadas para cada uma das colunas na tabela referente a este objeto no banco de dados, bem como suas respectivas funções set e get. 

class Car extends Model { 
    private $id;
    private $model;
    private $year;

    public function setId ($id) { ... };
    public function setModel ($model) { ... };
    public function setYear ($year) { ... };

    public function getId () { ... };
    public function getModel () { ... };
    public function getYear () { ... };
}

Ao definir as funções set de um model, é conveniente fazê-las retornar $this. Dessa forma, será possível agrupar as chamadas às funções set do objeto como no exemplo: 

public function setId ($id) { 
    $this->id = $id;
    return $this;
}

$object->setId ($id)->setName ($name)->setColor ($color)->setType ($type);

Chave primária

Por padrão, é considerado que a coluna de uma tabela no banco de dados usada como chave primária (primary key) chama-se "id".

Caso o nome da chave primária de uma tabela seja diferente, pode-se definir, no próprio model, a variável private $primarykey, a qual indica o nome do campo a ser usado para buscas por valores únicos na tabela de nome igual ao da classe. 

class Animal extends Model { 
    private $primarykey = 'animal_id'; 
}

Desta forma, ao relizar uma busca pela função get(), a consulta procurará pelo valor passado na coluna denominada "animal_id", ao invés de procurar por uma coluna chamada "id" e buscar tal valor nela.

Inicialização

Todos os models são carregados por padrão no carregamento da aplicação.

Todavia, caso haja a necessidade de incluir um model durante o desenvolvimento do projeto, a função model() se encarrega disso.

Após o model carregado, ressaltando, é preciso iniciar o objeto para fazer uso de suas funções. 

$object = new Bird;
$object->all ();

(new Bird)->all ();

toArray()

Converte os atributos e valores do objeto atual em um array, partindo dos métodos setters do objeto para a obtenção das informações. 

array toArray ( )

Usage example:
$device->toArray (); // Obtém um vetor contendo os atributos e valores do objeto $device

Retorna um vetor com os atributos do objeto.


jsonSerialize()

Converte os atributos e valores do objeto atual em uma string JSON. 

string jsonSerialize ( )

Usage example:
$guitar->jsonSerialize (); // Obtém a string JSON contendo os atributos e valores do objeto $guitar

Retorna uma string JSON com os atributos do objeto, partindo dos métodos setters do objeto para a obtenção das informações.



Banco de Dados

Para realizar a comunicação com o banco de dados é necessário que as constantes de banco de dados estejam corretamente definidas.

Todo model, quando extendendo o model principal "Model", herda as funções básicas de comunicação com o banco, definidos originalmente na classe DB.

Observação: para garantir o sucesso das consultas ao banco de dados, inclusive em funções como doLogin() e doRegister(), é importante que os vetores de dados sejam passados contendo em seu índice o nome do atributo da tabela a ser utilizado, conforme o exemplo: 

Data array:
$data = [
    'username' => $username,
    'password' => $password,
    'firstName' => $firstname,
    ...
];
 
Database table:
CREATE TABLE user (
    username VARCHAR(X) ... ,
    password VARCHAR(Y) ... ,
    firstName VARCHAR(Z) ... ,
    ...
);

Consultas
get()

Procura na tabela definida pelo objeto por um dado cuja chave primária seja igual a $id. 

object get ( $id )

Usage example:
(new Fruit)->get (5); // Procura na tabela Fruit pela primeira linha cujo valor da chave primária seja igual a 5
(new Product)->get ('rice'); // Procura na tabela Product pela primeira linha cujo valor da chave primária seja igual a 'rice

Retorna um objeto do tipo da classe que originou a chamada à função.


all()

Seleciona todos os dados da tabela definida pelo objeto. 

object_array all ( )

Usage example:
(new Juice)->all (); // Retorna um vetor de objetos do tipo Juice com todos os dados armazenados na tabela de mesmo nome

Retorna um vetor de objetos do tipo da classe que originou a chamada à função.


select()

Define o que deve ser buscado no banco de dados.

Quando executado sozinho junto à função find(), procura pelos dados recebidos por parâmetro (ou por todos os dados, quando o parâmetro não for recebido ou definido) sem uma condição de restrição (where) na tabela do objeto pelo qual a função é chamada. 

object:DB select ( [string $selectedData = NULL] )

Usage example:
(new Person)->select ('id, name, lastname')->find (); // Seleciona apenas 'id', 'name' e 'lastname' da tabela Person
(new Car)->select ('*')->find (); // Seleciona todos os dados da tabela Car
(new Animal)->select ()->find (); // Seleciona todos os dados da tabela Animal

Retorna um objeto da classe DB.

Observação: esta função deve ser executada SEMPRE seguida da função find(). Caso contrário, a consulta não será executada.


where()

Define a condição de restrição para a consulta dada pela função select() ou inicia uma nova busca por todas as colunas da tabela seguindo a condição de restrição passada.

A condição de busca ($where) é composta de uma string com o código SQL do where (exceto pela palavra chave "WHERE"), com os valores a serem buscados trocados por uma "?", enquanto que $whereVars pode ser uma string contendo o único valor a substituir a "?" na string $where ou um vetor com todos os elementos que substituirão, respectivamente, cada uma das interrogações do código passado. 

object:DB where ( string $where, $whereVars )

Usage example:
(new Cat)->where ('name = ?', 'Felix')->find (); // Procura por toda a tabela Cat por um valor cujo atributo 'name' seja 'Felix'
(new Dog)->select ('id, age')->where ('name = ? AND weight > ?', [$name, $weight])->find (); // Seleciona 'id' e 'age' na tabela Dog, procurando por uma linha em que 'name' seja igual ao valor de $name e 'weight' seja maior que o valor de $weight

Retorna um objeto da classe DB.

Observação: esta função deve ser executada SEMPRE depois da função select() (quando aplicável) e seguida da função find(). Caso contrário, a consulta não será executada.


orderBy()

Define o campo pelo qual os resultados da consulta serão ordenados em $orderValue e a forma de ordenação dos dados (ASC ou DESC) em $order.

Pode ser usada várias vezes em uma mesma consulta para definir os vários campos pelos quais a consulta deverá ordenar os dados. 

object:DB orderBy ( string $orderValue, [string $order = 'ASC'] )

Usage example:
(new People)->select ()->orderBy ('name', 'DESC')->find (); // Ordena o resultado da busca pelo campo 'name' de forma descendente (DESC)
(new Dog)->select ('id, name')->where ('weight > ?', $weight)->orderBy ('name')->orderBy ('weight', 'DESC')->find (); // Ordena os resultados pelo atributo 'name' de forma ascendente e, posteriormente, pelo atributo 'weight' de forma descendente (DESC)

Retorna um objeto da classe DB.

Observação: esta função deve ser executada SEMPRE após as funções select() e where(), em qualquer situação, e seguida da função find(). Caso contrário, a consulta não será executada.


extraSql()

Adiciona código SQL à consulta.

Caso seja necessário adicionar código SQL à consulta, como um JOIN ou outro comando, a função realiza este procedimento antes da execução da consulta. 

object:DB extraSql ( string $sql )

Usage example:
(new Television)->select ()->extraSql ('Some SQL')->find (); // Acrescenta o SQL à consulta
(new Printer)->select ('id, code')->where ('code < ?', $code)->orderBy ('name')->extraSql ('Another SQL')->find ();

Retorna um objeto da classe DB.

Observação: esta função deve ser executada SEMPRE após as funções select() ou where(), em qualquer situação, e seguida da função find(). Caso contrário, a consulta não será executada.

Autenticação

Para facilitar o processo de autenticação e simplificar o sistema de login e registro de usuários, há a classe Auth.

Todas as funções dessa classe são estáticas.

Observação: ao utilizar esta classe, deve-se lembrar que o campo destinado à senha do usuário é criptografado dentro das funções nativas da classe. Para que seus métodos funcionem corretamente, ao criar um model para o objeto usuário é preciso definir que na função setPassword() (ou equivalente destinada a preencher o campo da senha) a atribuição ao valor será feita através da função hashPassword(). 

public function setPassword ($password) {
    $this->password = Auth::hashPassword($password);
    return $this;
}

Auth::doLogin()

Recebe os dados do usuário $authData e realiza o login, retornando à página anterior e enviando os erros encontrados e os valores dos campos (vide withErrors() e withValues()) para que sejam exibidos ao usuário, ou realizando a criação da sessão (vide createAuthSession()) para o usuário, redirecionando-o para uma página especificada por $redirectLocation (ou para a página inicial, caso não seja passado o valor) e buscando os dados na tabela $dbTable (definida como 'User' caso não seja passada) para fazer a validação.

Observação: essa função utiliza o índice do vetor passado como parâmetro para identificar o atributo na tabela do banco de dados a ser comparado. Por isso, é essencial que o formulário tenha em seus campos o atributo name igual ao nome das suas respectivas colunas no banco, conforme citado em Banco de Dados. 

bool doLogin ( array $authData, [string $redirectLocation = NULL], [string $dbTable = 'User'] )

Usage example:
Auth::doLogin (filterPost(), 'logged/home', 'Users');
Auth::doLogin ($data); // Ao final redireciona para a raiz (/) e procura os dados na tabela 'User'

Retorna TRUE caso o login seja bem sucedido ou FALSE caso contrário.

Redefinição de senha

Há também um conjunto de funções criadas para facilitar o processo de redefinição de senha do usuário. Estas funções exigem que o banco de dados possua algumas características.

Para que funcionem as funções, é preciso que o banco de dados tenha uma tabela específica para armazenar as requisições de redefinição de senha. Essa tabela deve conter, no mínimo, três colunas, sendo elas email (referente ao e-mail do usuário, que deve estar disponível na tabela de usuários), token (o código alfanumérico de redefinição gerado automaticamente) e val (indica se o código já foi usado ou não). É um sistema bastante simples, que não envolve tempo para expiração de código ou métodos mais sofisticados. 

CREATE TABLE passwordreset (
    id INT ... ,
    email VARCHAR ... ,
    token VARCHAR ... ,
    val BOOL ... 
);

Esta tabela é o suficiente para que o mecanismo de recuperação de senha funcione.

Observação: o nome das três colunas citadas DEVE ser exatamente como descrito. Caso contrário, a redefinição de senha não funcionará.


Auth::requestPassReset()

Função que recebe o e-mail do usuário pela variável $email, o nome da classe de usuários pela variável $userClass, o nome da classe de requisições de redefinição de senha pelo parâmetro $resetRequestClass e a URL para onde o usuário será mandado ao receber o e-mail com o link para a redefinição por $urlValidate e gera o token de redefinição de senha do usuário, o registra e envia por e-mail.

É possível também que seja passada a variável $mailContent para a função, contendo o conteúdo (em texto ou HTML) da mensagem que será enviada ao usuário junto com o link - que será anexado ao final da mensagem automaticamente. 

bool requestPassReset ( string $email, string $userClass, string $resetRequestClass, string $urlValidate, [string $mailContent] )

Usage example:
Auth::requestPassReset ($email, 'User', 'PasswordReset', 'user/resetpassword');
Auth::requestPassReset ($email, 'Users', 'NewPassRequest', 'resetpassword', $emailContent);

Retorna TRUE caso tenha sucesso ao enviar o e-mail ou FALSE caso contrário.


Auth::checkToken()

Função que recebe o token de redefinição $token (passado para o método do controller como parâmetro get na rota) que o usuário recebeu no e-mail e o nome da classe de requisições de redefinição de senha pelo parâmetro $resetRequestClass e verifica se o token está correto, comparando-o com os registros do banco e analisando sua validade. 

bool checkToken ( string $token, string $resetRequestClass )

Usage example:
Auth::checkToken ($token, 'PasswordReset');
Auth::checkToken ($dataGET['id'], 'PasswordReset'); // Recebe o nome de 'id' ao ser passado da rota para o método por ser o primeiro parâmetro

Retorna TRUE caso os dados confiram ou FALSE caso não.


Auth::resendEmail()

Função que recebe o e-mail do usuário pela variável $email, o nome da classe de usuários pela variável $userClass, o nome da classe de requisições de redefinição de senha pelo parâmetro $resetRequestClass e a URL para onde o usuário será mandado ao receber o e-mail com o link para a redefinição por $urlValidate, obtém os dados de redefinição salvos e reenvia o e-mail ao usuário.

É possível também que seja passada a variável $mailContent para a função, contendo o conteúdo (em texto ou HTML) da mensagem que será enviada ao usuário junto com o link - que será anexado ao final da mensagem automaticamente. 

bool resendEmail ( string $email, string $userClass, string $resetRequestClass, string $urlValidate, [string $mailContent] )

Usage example:
Auth::resendEmail ($email, 'User', 'PasswordReset', 'user/resetpassword');
Auth::resendEmail ($email, 'Users', 'NewPassRequest', 'resetpassword', $emailContent);

Retorna TRUE caso tenha sucesso ao enviar o e-mail ou FALSE caso contrário.


Auth::changePassword()

Função que recebe o array contendo a nova senha e a sua confirmação através da variável $passData, realiza a validação dos valores (verifica se ambas estão preenchidas e são iguais), recebe o token de redefinição pela variável $token, o nome da classe de usuários em $userClass e o nome da classe de requisições de redefinição em $resetRequestClass e verifica se o token (que pode ser enviado para o método do controller por meio de um campo oculto no formulário) está correto e altera a senha do usuário.

Essa função também é responsável por invalidar o token utilizado, para que ele não seja usado duas vezes. 

bool changePassword ( array $passData, string $token, string $userClass, string $resetRequestClass )

Usage example:
Auth::changePassword (filterPost(), $token, 'User', 'PasswordReset');
Auth::changePassword (['password' => $data['password'], 'password_confirm' => $data['password_confirm']], $token, 'User', 'PasswordReset');

Retorna TRUE caso tenha sucesso ao alterar a senha do usuário ou FALSE caso contrário.

withValues()

		

			Envia um flash com os valores dos campos de formulário junto a um 
			redirect() ou back() para que os campos
			possam ser automaticamente preenchidos com os valores digitados antes do último envio
			do formulário.
		

		

			Seu valor deve ser exibido no formulário com a função 
			oldVal().
		

		

			Observação: esta função pode ser utilizada 
			APENAS EM CONJUNTO com as funções 
			redirect() e back().
		

		
			
object:App withValues ( )

			

Usage example:

			
back()->withValues();

			
redirect('login/form')->withValues();

		
		

			Retorna um objeto da classe App.
		

		

			Função relacionada: oldVal()
		

		

		
withErrors()

		

			Envia um flash com as mensagens de erro referentes aos campos de formulário 
			junto a um redirect() ou back() para o
			usuário veja 
		

		

			Seu valor deve ser exibido no formulário com a função 
			getInputErrors().
		

		

			Observação: esta função pode ser utilizada 
			APENAS EM CONJUNTO com as funções 
			redirect() e back().
		

		
			
object:App withErrors ( )

			

Usage example:

			
back()->withErrors();

			
redirect('register')->withErrors();

		
		

			Retorna um objeto da classe App.
		

		

			Função relacionada: getInputErrors()
		



		


		

		
Validação de Formulários

		
Validation::check()

		

			Função estática da classe Validation. 
		
 
		

			Checka a validade dos valores em $data, normalmente o vetor resultante do
			recebimento de um formulário, com base em regras de validação passadas em $rules
			e redireciona de volta ao formulário, carregando em flash os erros encontrados
			em cada campo.
		

		
			
bool Validation::check ( array $data, array $rules )

			

Usage example:

			
Validation::check (

			
   filterPost(),

			
   array(

			
      'username' => 'required|min:6|alphanum|max:12',

			
      'email' => 'required|min:6|email',

			
      'password' => 'required|equal:confirm_password',

			
      'confirm_password' => 'required',

			
   )

			
);

		
		

			Redirecione e retorna TRUE caso não encontre erros e FALSE caso contrário.
		

		

		
Opções de validação

		
			
required // Campos obrigatórios

			
int // Números inteiros

			
decimal // Números decimais

			
alpha // Apenas letras e espaço

			
alphanum // Apenas letras, números e espaço

			
email // E-mail com sintaxe válida

			
min:X // Tamanho mínimo de caracteres definido por X

			
max:X // Tamanho máximo de caracteres definido por X

			
equal:Y // Deve ser igual ao campo Y

		


		


		

		
Redirecionamento

		
redirect()

		

			Redireciona o visitante a outra página, definida por $newAddress (com base no
			endereço da raiz da aplicação para endereços internos), com o código HTTP definido 
			por $statusCode.
		
 
		

			Caso não seja passado o endereço, redireciona para a
			página inicial da aplicação. Caso não seja passado o código HTTP, utiliza o padrão 200.
		

		
			
object:App redirect ( [string $newAddress = ''], [int $statusCode = 200] )

			

Usage example:

			
redirect ('posts');

			
redirect ('error/notfound', 404);

			
redirect (); // Redireciona para a página inicial

		
		

			Retorna um objeto da classe App.
		

		

		
back()

		

			Redireciona o visitante para a página anterior.
		

		
			
object:App back ( )

			

Usage example:

			
back ();

		
		

			Retorna um objeto da classe App.
		


		

		
		
Uso conjunto

		

			Tanto a função redirect() quanto a função back()
			podem ser usadas em conjunto com algumas outras funções:
		

		

		

			Redirecionar retornando o preenchimento do formulário	
			
				
back()->withValues();

				
redirect('login/form')->withValues();

			
			

				Lembre-se: para exibir os dados retornados, é preciso usar a função
				oldVal() no formulário.
			

			

				Referência: withValues()
			

			


			Redirecionar retornando os erros encontrados no formulário		
			
				
back()->withErrors();

				
redirect('register')->withErrors();

			
			

				Lembre-se: para exibir os erros retornados, é preciso usar a função
				getInputErrors() no formulário.
			

			

				Referência: withErrors()
			

			


			Redirecionar retornando o preenchimento do formulário e os erros encontrados
			
				
back()->withValues()->withErrors();

				
redirect('some/form')->withErrors()->withValues();

			
			

				Lembre-se: para exibir os dados e erros retornados, é preciso usar as funções
				oldVal() e getInputErrors() 
				no formulário.
			

			

				Referência: withErrors() e 
				withValues()
			

			


			Redirecionar definindo uma mensagem flash
			
				
back()->flash('itsaflash', 'some message');

				
redirect()->flash('itsaflash', 'some message');

			
			

				Lembre-se: para exibir o valor da variável flash, você precisa exibí-lo
				na página destino com o uso da função flash(). Seu conteúdo será
				apagado em seguida.
			

			

				Referência: flash()
			

			


			Redirecionar definindo uma variável de sessão
			
				
back()->session('itsasession', 'some content');

				
redirect('example')->session('itsasession', 'some content');

			
			

				Lembre-se: para exibir o valor da variável de sessão, você precisa exibí-lo
				na página destino com o uso da função session().
			

			

				Referência: session()
			

			


			Redirecionar definindo um cookie
			
				
back()->cookie('itsacookie', 'other content');

				
redirect('somewhere')->cookie('itsacookie', 'other content');

			
			

				Lembre-se: para exibir o valor do cookie, você precisa exibí-lo
				na página destino com o uso da função cookie().
			

			

				Referência: cookie()
			

			


			Redirecionar destruindo um cookie
			
				
back()->unsetcookie('itsacookie');

				
redirect('another/place')->unsetcookie('itsacookie');

			
			

				Referência: unsetcookie()
			

		



		


		

		
Funções utilitárias
		
		
dump()

		

			Realiza o despejo (dump) de uma variável ou retorno de uma função. É basicamente a
			função var_dump(), nativa do PHP, com sua visualização otimizada.
		

		
			
void dump ( $var or function() )

			

Usage example:

			
dump ($cars);

			
dump (getCars());

		
		

			Não possui retorno, apenas exibe os dados.
		

		

		
dd()

		

			Realiza o despejo (dump) de uma variável ou retorno de uma função e encerra a execução
			do script. É a abreviação de "Dump and Die".
		
		
		
			
void dd ( $var or function() )

			

Usage example:

			
dd ($cars);

			
dd (getCars());

		
		

			Não possui retorno, apenas exibe os dados.
